# 山海商店目录性能与按需同步设计

> 作者：山海恒长在 / dishanhai

## 目标

在不改变现有购买、出售、分类、搜索、隐藏跳转、奖励选择、编辑、限购和钱包交互语义的前提下，从数据加载、客户端同步、索引和可视区域渲染四个层面解决大量商品导致的首次打开卡顿、持续帧开销和内存占用问题。

本设计同时修复当前仅在单机共享 JVM 中偶然成立的目录同步行为，使单机和独立服务器统一使用服务端权威目录。

## 当前问题与规模

当前游戏端 `config/gt_shanhai/shop.json` 约 197 KiB，共 186 条商品：

- “无限盘区”单个主分类包含 130 条商品。
- 145 条商品带 NBT。
- 154 条商品带自定义显示图标。
- 首屏在当前布局下可能同时出现约 80～100 个物品模型。

现有链路存在以下问题：

1. `ShopOpenPacket` 只发送编辑权限，不发送商品目录。
2. `ShopRefreshPacket` 是零载荷通知，只调用 `resize()`，不会同步服务端数据。
3. 独立服务器客户端会在首次打开时从客户端本地 `shop.json` 同步读盘和解析；单机看似正常只是逻辑服务端与客户端共享静态 `ShopConfig.ENTRIES`。
4. `ShopConfig.reload()` 先清空可变全局列表再逐条填充，不能安全地直接放入后台线程。
5. 界面虽然在循环内跳过不可见商品，但渲染、点击和 tooltip 仍从头遍历全部结果，属于“伪虚拟化”。
6. 搜索每输入一个字符就重建全部商品的名称和组合成分文本。
7. `linkTo`、分类、客户端发送的全局索引等操作仍依赖线性扫描或客户端与服务端列表顺序完全一致。
8. 编辑和有限次数商品成交会在服务端线程同步重写整个 `shop.json`。

## 总体架构

采用四层结构：

```text
后台 I/O/纯 DTO 解析
        ↓
服务端权威 CatalogSnapshot（revision + 预计算索引）
        ↓  manifest / 按需分块 / 状态增量
客户端 ClientShopCatalog（版本缓存 + 分类 LRU + 请求代次）
        ↓
ShopScreen（主线程预算实体化 + 可视区虚拟化）
```

所有结构性目录变更都产生新的 `catalogRevision`。客户端只接受当前 revision 和当前请求代次的数据，迟到的旧分块不得覆盖新目录。

## 服务端权威目录快照

新增结构性不可变的 `ShopCatalogSnapshot`，至少包含：

- `long revision`
- 按服务端顺序排列的商品定义
- `entryKey -> 商品定义` 的 O(1) 索引
- 主分类和子分类清单
- `主/子分类 -> entryKey 顺序列表`
- `linkKey -> entryKey`
- 用于客户端轻量搜索清单的商品 ID、分类、自定义名称和组合商品 ID
- 每个分类的条目数及分块元数据

`ShopConfig` 不再向读取方暴露可变 `ArrayList` 的实时包装，而是完整构建新快照后一次性替换引用。读取方只能看到完整旧快照或完整新快照，不会观察到空目录或半加载目录。

### 商品身份

每个快照为商品分配服务端 `entryKey`。所有购买、出售、删除、编辑和奖励选择请求改为携带：

```text
catalogRevision + entryKey
```

服务端只从当前权威快照查找商品。revision 过期时拒绝操作、同步新 manifest 并提示客户端刷新，不再回退到可能碰撞的 `goodsId + category`，也不再依赖客户端局部列表的 `indexOf()`。

`entryKey` 只要求在同一 revision 内稳定，不写入 `shop.json`；结构变化后由 revision 负责隔离旧身份。

### 运行状态与结构版本分离

永久总限购剩余次数等运行状态不应让整个目录结构 revision 每次成交都变化。目录定义与运行状态分开：

- 新增、删除、编辑、重载：发布新结构 revision。
- 成交扣减剩余次数：发送按 `revision + entryKey` 定位的状态增量。
- 客户端缓存条目定义不因一次成交全部失效。

## 异步加载与线程边界

项目复用现有“单后台 worker + Future + 主线程轮询提交 + revision 发布”模式。

### 允许后台执行

- 文件读取。
- Gson 解析为不访问 Minecraft API 的纯 DTO。
- 基础字段、大小和数量限制校验。
- 纯字符串搜索文档预处理。
- 快照持久化内容的 Gson 序列化。
- 临时文件写入和原子替换。

### 必须留在 Minecraft 主线程

- Forge 注册表查询。
- `ItemStack`、物品能力、tooltip 和本地化名称访问。
- `GuiGraphics`、`RenderSystem`、物品模型和流体贴图渲染。
- 服务端目录原子发布及交易结算。
- 客户端最终目录状态发布。

后台解析完成后，服务端线程把 DTO 实体化为有效商品定义并发布新快照。解析或实体化失败时保留最后一个可用快照，绝不能先清空当前目录。

首次开服可在 `ServerAboutToStartEvent` 启动解析，并由后续服务端 tick 轮询提交；目录尚未就绪时打开请求返回明确的“商店目录加载中”状态。运行期重载继续服务旧快照，直到新快照完整发布。

## 按需目录同步

客户端新增独立的 `ClientShopCatalog`，`ShopScreen` 不再直接读取 `ShopConfig` 或本地 `shop.json`。单机也必须经过同一套网络协议，消除跨逻辑侧静态列表共享和线程竞争。

### 打开流程

```text
玩家请求打开钱包
  → 服务端发送 canEdit + catalogRevision + manifest
  → 客户端立即打开商店壳层
  → 客户端请求当前主/子分类所需分块
  → 分块到达后进入主线程预算队列
  → 商品卡片渐进替换加载占位
```

manifest 包含：

- 目录 revision。
- 主/子分类层级及各组商品数。
- 轻量搜索条目：`entryKey`、分类、商品注册 ID、组合商品注册 ID、自定义显示名。
- 分组分块数量和必要的跳转目标定位信息。

manifest 不携带大 NBT、奖励池和完整图标栈。全局搜索先在轻量清单中定位 `entryKey`，再按结果请求完整条目，从而保留跨分类搜索而不预载全部商品内容。

### 分块协议

新增请求与响应包，至少携带：

```text
revision
requestId
group/search 标识
chunkIndex
chunkCount
entry payload
```

分块以编码后字节预算为主，而不是只按条目数量切分，因为商品 NBT、奖励池和自定义图标大小差异很大。每包设置严格的条目数、字符串、NBT 和解码后字节上限；新消息只追加到现有注册顺序末尾，并提升网络协议版本。

客户端只接受：

- revision 与当前 manifest 一致；
- requestId 与当前分类或搜索请求一致；
- chunkIndex 合法且未重复；
- 总块数和负载大小在硬上限内。

### 缓存与预取

- 同一服务器连接、同一 revision 下重复打开直接复用已加载分类，不重复同步。
- 当前分类优先请求；接近已加载尾部时预取下一块。
- 相邻分类可在空闲时低优先级预取。
- 完整分类采用有上限的 LRU 缓存，避免玩家浏览一遍后仍把所有大型 NBT 长期常驻。
- 切换服务器、断线或 revision 改变时清理不兼容缓存。

### 编辑与刷新

`ShopRefreshPacket` 改为明确的目录失效通知，至少包含新 revision 和受影响分类。新增、删除、编辑或 `/商店 reload` 后应通知所有相关在线客户端，而不是只刷新操作发起者。

收到新 revision 后：

- 取消旧请求。
- 丢弃迟到分块。
- 重新绑定或清空旧 `selected`。
- 保留与新 revision 明确兼容的界面状态，不再通过 `resize()` 假装同步。

## 客户端主线程预算加载

网络线程只解码纯传输 DTO。`ItemStack`、本地化名称、显示图标和 tooltip 稳定部分在客户端主线程按预算实体化。

默认每帧使用约 1～2 ms 或固定小批量预算，优先级为：

1. 当前可视区。
2. 可视区前后各一行 overscan。
3. 当前分类剩余条目。
4. 预取分类和轻量搜索缓存。

未实体化条目显示轻量加载占位，不在同一帧强行创建约百个带 NBT 的 `ItemStack`。关闭界面或切换请求时通过 generation/requestId 取消过期队列。

不缓存整张物品渲染纹理，因为附魔 glint、耐久条、动态 model override 和动画贴图会失真。

## 可视区域渲染优化

### 真正的行虚拟化

根据 `scroll`、`ROW_STRIDE` 和 `contentHeight()` 直接计算首尾可见行，只遍历可见行和一行 overscan。`drawGrid()` 的复杂度从当前结果集大小 `O(N)` 降为可见商品数 `O(V)`。

### O(1) 命中计算

新增统一的 `entryIndexAt(mouseX, mouseY)`：

- 先判断鼠标是否位于网格矩形。
- 通过局部坐标和滚动量计算行、列。
- 排除 1 像素格间隙。
- 直接得到 `row * columns + column`。

点击和 tooltip 共用这一方法，不再各自遍历全部商品，也避免三套坐标算法漂移。

### 稳定显示缓存

扩展现有单格缓存，缓存：

- 完整显示名和截断显示名。
- 主商品栈或自定义/组合显示图标列表。
- 稳定成本图标及文本。
- tooltip 中不随余额变化的成分文本。
- 轻量搜索文本。

缓存设置容量上限，并在目录 revision、语言或资源重载时失效。钱包余额、背包数量和可购买状态只能缓存到当前客户端 tick，不能长期缓存。

### 搜索与其他逐帧分配

- 搜索输入采用约 100～150 ms 防抖。
- 查询字符串只标准化一次。
- 商品搜索文档只在目录或语言变化时重建。
- `linkKey` 走快照 O(1) 索引。
- 货币栏排序结果按钱包/目录 revision 缓存。
- 背包持有量和可购买判定在同一客户端 tick 内复用。

## 异步持久化

结构编辑和限购状态保存使用单线程串行队列：

1. 服务端线程生成不可变的待保存数据快照并递增保存 revision。
2. 后台线程只保留尚未落盘的最新 revision，合并短时间内的重复保存请求。
3. 写入同目录临时文件。
4. 完整写入并关闭后使用原子移动替换正式文件；平台不支持原子移动时使用受控回退。
5. 服务端停止时等待最后一次保存完成，并设置有界超时和错误日志。

这样有限次数商品连续成交不会在每个服务端 tick 中多次全量覆写约 200 KiB 文件，同时避免直接覆盖时崩溃留下截断 JSON。

## 兼容与边界行为

- 旧 `shop.json` 无需手工迁移。
- 目录解析失败继续服务最后一个可用快照。
- 客户端缓存过期时拒绝交易并自动刷新，不能猜测对应条目。
- 隐藏商品仍可通过合法 `linkTo` 请求并显示，但不进入普通分类列表。
- 搜索仍覆盖全部分类，并保留物品 ID、自定义名称和组合商品成分匹配。
- 编辑商品移动分类时同时失效旧分类和新分类。
- 奖励选择和编辑界面改用 `revision + entryKey`，不再计算全局列表下标。
- 单机与独立服务器行为一致。
- 新网络包必须声明方向、限制集合大小，并提升通道协议版本。
- 本次不改变商店视觉布局，不通过减少可见商品数量掩盖性能问题。
- 本次不修改非山海署名文件，也不涉及 KubeJS/Rhino 脚本。

## 实施阶段

1. 提取纯范围计算、请求版本和快照索引模型，先建立单元测试。
2. 建立服务端快照、revision、O(1) 分类/链接/商品身份索引。
3. 建立 manifest 和完整条目分块协议，客户端停止读取本地 `shop.json`。
4. 将交易、奖励选择和编辑链路迁移到 `revision + entryKey`。
5. 将 `ShopScreen` 接入客户端目录缓存、加载占位、分帧实体化和真虚拟化。
6. 加入搜索防抖、稳定显示缓存及每 tick 动态状态复用。
7. 加入串行合并保存和原子替换。
8. 删除旧的共享列表假同步路径，并完成构建与游戏端验证。

每一阶段均保持可单独编译和验证，避免把协议、界面和持久化一次性混成不可回滚的大改。

## 验证标准

### 自动验证

1. 快照只会完整原子发布，后台失败时旧快照仍可读取。
2. 分类、子分类、隐藏条目、链接和 entryKey 索引结果与原行为一致。
3. 旧 revision、旧 requestId、重复块、越界块和超限负载均被拒绝。
4. 分块乱序到达能正确组装，同 revision 重复打开不重复请求已缓存分类。
5. `revision + entryKey` 能精确定位同物品、同分类但 NBT 不同的多条商品。
6. 可视区范围计算覆盖顶部、中间、底部、部分可见行和网格间隙。
7. 点击和 tooltip 使用同一 O(1) 命中结果。
8. 搜索防抖只发布最后一次查询结果，旧查询不能覆盖新查询。
9. 合并保存只落盘最新 revision，临时文件失败不破坏正式文件。
10. 当前 Gradle `test`、`check` 和完整 `build` 全部通过。

### 游戏端验证

1. 首次打开立即出现商店壳层，不再由全目录同步解析阻塞一个长帧。
2. 当前分类商品渐进出现，滚动到未加载区域会继续请求而不冻结界面。
3. 同 revision 关闭后重开不重复同步已缓存目录。
4. 大分类持续渲染时，商品总数增加不会让点击、tooltip 和逐帧遍历按全量线性增长。
5. 分类切换、全局搜索、隐藏跳转、购买、出售、奖励选择和编辑行为保持一致。
6. 编辑或重载后所有在线客户端收到新 revision，旧分块不会回写新目录。
7. 同物品不同 NBT 的商品始终交易正确条目。
8. 连续限购成交不再每次同步全量覆写目录，停服后最终状态仍能落盘恢复。
9. 单机和独立服务器均由服务端权威目录驱动，客户端本地文件差异不影响商品内容。

## 性能判定原则

不以“代码可编译”或“小规模目录看起来不卡”作为性能完成证据。实现时应增加只在调试开关下启用的分阶段计时，至少区分：

- manifest 与分块解码时间；
- 客户端主线程实体化预算时间；
- `drawGrid` 可见条目数与耗时；
- 搜索索引与过滤耗时；
- 保存队列序列化和磁盘写入耗时。

最终结论需结合算法复杂度测试、构建结果和当前 186 条真实目录的游戏端实测，不能把异步任务转移后产生的乱序、半状态或错误交易当成性能优化。
