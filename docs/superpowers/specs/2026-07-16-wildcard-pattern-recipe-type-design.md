# 星律通配符样板按槽分配配方类型设计

## 目标

在星律样板总成的通配符配置页中，让玩家从当前连接主机支持的配方类型中选择一个类型，并把它分配给指定的通配符母槽。五个母槽独立配置，类型写入通配符样板物品自身 NBT。

## 交互

- 保留现有五个通配符母槽和展开样板预览。
- 增加 `1-5` 槽位选择控件，当前选择明确高亮。
- 增加带滚动条的配方类型列表。
- 列表来源为星律当前连接的全部配方逻辑主机所暴露的 `getRecipeTypes()`，按注册 ID 去重。
- 玩家点击某个配方类型后，将该类型写入当前选中母槽里的通配符样板。
- 提供“自动识别”选项，用于删除手动类型标记并恢复现有自动推断行为。
- 空槽不可分配。已分配类型若不再属于当前主机，仍保留 NBT，并在界面中显示为当前不可用。

## 数据模型

- 复用 `PatternRecipeTypeHelper.TAG_RECIPE_TYPE`，不新增重复 NBT 字段。
- 类型保存在母样板 ItemStack 中，因此样板被取出、移动或放入另一台星律后仍保留分配。
- 星律机器本身只保存当前 UI 选中的母槽序号；配方类型不另存一份机器侧映射。
- 旧通配符样板没有类型字段时继续使用自动推断，保持向后兼容。

## 展开与执行

1. 星律读取母样板上的配方类型 ID。
2. 通配符模组照常生成材料展开结果。
3. 未指定类型时，沿用现有全类型配方匹配。
4. 已指定类型时，只在该 `GTRecipeType` 内匹配展开结果。
5. 匹配不到指定类型配方的展开结果直接丢弃。
6. 匹配成功后，将同一类型标记写入动态编码样板，继续走星律现有缓存、供料和虚拟执行链路。

## 代码边界

- 只修改 `gt_shanhai` 内山海署名文件。
- 不修改 `lib_source/[WP]wildcard_pattern-0.1.2-gtl`，该目录仅作为行为参考。
- 不修改通配符模组的手持配置界面。
- UI、NBT 读写和展开过滤集中在星律通配符兼容边界，不扩散到普通 AE 样板。

## 异常与兼容

- 未连接主机：滚动列表显示无可用类型，不修改已有分配。
- 多主机连接：合并所有主机类型并按注册 ID 去重。
- 类型已注销或无法解析：保留原始 NBT，展开结果为空，避免静默改成其他类型。
- 母样板配置或类型变化：重建对应动态槽并清理旧配方缓存，现有内部物料按既有退款路径处理。
- GTLCore 的 `DUMMY_RECIPES` 辅助输入仍绕过槽位类型过滤，且不清理主配方虚拟目标。

## 验证

- 单元测试覆盖 NBT 类型读写、自动识别清除、指定类型过滤和无匹配结果。
- 集成测试覆盖五个母槽可独立分配不同类型，并确认动态槽继承母槽类型。
- 回归测试覆盖普通星律样板、未标记通配符样板和反熵冷凝 dummy 输入。
- 完整执行 `gradle build --no-daemon`，确认全部测试通过并产出 Jar。
