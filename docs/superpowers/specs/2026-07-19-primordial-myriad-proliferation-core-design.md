# 原初万象衍生核心设计规格

## 目标

新增一个只能挂载到原始终焉引擎的功能型原初模块“原初万象衍生核心”。核心形成并连接宿主后，为宿主挂载的所有配方加工模块提供阶梯式产出增益。增益不叠加，多个核心同时存在时只取当前最高有效阶段。

## 命名

- 方块与机器名称：原初万象衍生核心
- 英文名称：Primordial Myriad Proliferation Core
- 注册 ID：`gt_shanhai:primordial_myriad_proliferation_core`
- Java 类名：`PrimordialMyriadProliferationCore`
- 推荐描述：万象自一而生，一念衍化千重。

## 晋升阶段

| 核心状态 | 宿主配方产出倍率 |
| --- | ---: |
| 未安装、结构失效或未连接宿主 | 1x |
| 已形成并连接宿主，但未运行指定晋升配方 | 10x |
| 正在运行二级晋升配方 | 100x |
| 正在运行一级晋升配方 | 1000x |

“正在运行”以核心自身 `RecipeLogic` 的实际工作状态为准。配方暂停、断能、完成、中断或不再处于工作状态时，倍率立即退回 10x。

## 专属配方类型

创建两个 `gtceu` 命名空间下的专属配方类型：

### 二级晋升

- ID：`gtceu:primordial_myriad_ascension_tier_2`
- 中文名：原初万象衍生·二级晋升
- 有效倍率：100x
- 最大物品输入：4
- 最大物品输出：0
- 最大流体输入：4
- 最大流体输出：0

### 一级晋升

- ID：`gtceu:primordial_myriad_ascension_tier_1`
- 中文名：原初万象衍生·一级晋升
- 有效倍率：1000x
- 最大物品输入：4
- 最大物品输出：0
- 最大流体输入：4
- 最大流体输出：0

两个配方类型均为耗能多方块配方。配方本身没有输出，只有在真实匹配输入、扣除输入并持续推进进度时才提供高阶段倍率。具体配方内容由后续 KubeJS 配方脚本定义，本模组只提供类型和运行语义。

### 注册时机

两个类型由九周目游戏端山海署名的 startup script 在 `GTCEuStartupEvents.registry('gtceu:recipe_type')` 中注册，脚本保持 IIFE 包裹并只使用 Rhino 兼容语法。Java 侧禁止再次调用 `GTRecipeTypes.register` 创建同名类型，只通过 `GTRegistries.RECIPE_TYPES.get(new ResourceLocation('gtceu', id))` 查表引用，确保 KubeJS 启动期和 JEI 都能稳定看见类型。

若任一类型在机器注册阶段仍未找到，机器注册必须明确记录缺失 ID 并拒绝用其他配方类型替代，避免一级、二级阶段被错误映射。

## 模块结构与注册

核心继承现有 `PrimordialOmegaEngineModuleBase`，沿用原初模块的宿主连接、无线能源、生命周期和标准反基督锻炉模块结构。结构允许物品与流体输入能力，不要求任何输出能力。

机器定义同时绑定两个专属配方类型。核心不允许脱离宿主向其他机器提供增益；即使独立工作配置允许模块自身形成，未连接原始终焉引擎时倍率也保持 1x。

## 宿主倍率聚合

新增窄接口 `IPrimordialOutputMultiplierModule`，只暴露当前有效产出倍率。原初万象衍生核心实现该接口。

`PrimordialOmegaEngineMachine` 在已连接模块集合中查找该接口并取最大值：

1. 没有有效核心时返回 1。
2. 任意空闲有效核心至少提供 10。
3. 任意核心正在运行二级晋升配方时至少提供 100。
4. 任意核心正在运行一级晋升配方时返回 1000。
5. 达到 1000 后立即停止扫描。

结果按宿主 tick 缓存；模块添加、移除和结构失效时主动失效缓存。该设计避免每个并行配方重复遍历模块集合，同时能在晋升配方状态变化后的下一个 tick 立即更新。

多个核心不进行乘法或加法叠加。一个空闲核心和一个一级运行核心同时存在时，最终倍率仍为 1000x。

## 配方输出变换

输出增益只作用于使用 `PrimordialModuleRecipeLogic` 的已挂载加工模块，不修改原始终焉引擎自身，也不放大直接 Tick 产电等非配方产出。

在原初模块统一配方逻辑中增加单一输出变换入口：

1. 从模块宿主读取当前倍率。
2. 倍率为 1 时直接返回原配方。
3. 使用 `ContentModifier.multiplier(multiplier)` 创建配方副本，只修改输出，不修改输入、时长、EU/t 或并行。
4. 将副本的 `outputs` 与 `tickOutputs` 中所有 `Content` 的 `chance` 和 `maxChance` 设为 10000。
5. 在概率抽取前使用该副本，因此物品、流体和其他配方输出能力全部按绝对 100% 概率结算。

变换必须覆盖两条无线配方构建路径：实时处理的 `shouldProcess` 路径和已有 `processedRecipeList` 路径。每条路径只放大一次，禁止对已经放大的副本再次乘倍率。

## 输出容量与资源消耗

`getMaxParallel` 和最终输出匹配必须使用放大后的输出副本，以防原始输出能够放入、放大输出却溢出。输入匹配和输入扣除继续使用原配方，因此 10x、100x、1000x 都不会放大输入成本。

配方 EU、持续时间和并行计算仍使用原始配方。晋升核心自身运行的晋升配方正常消耗其输入和无线能源。

## 状态显示

核心机器界面与信息文本显示：

- 是否连接原始终焉引擎。
- 当前阶段：基础衍生 / 二级晋升 / 一级晋升。
- 当前宿主产出倍率：10x / 100x / 1000x。
- 当前运行的晋升配方类型。

原始终焉引擎信息文本增加当前最高“万象衍生倍率”，便于确认多个核心取最高值而没有叠加。

## 异常与边界行为

- 核心结构失效或从宿主移除：宿主在同 tick 失效缓存，后续配方恢复正常倍率或由其他核心决定。
- 晋升配方暂停或断能：不视为正在运行，立即退回 10x。
- 多核心不同阶段：只取最大倍率。
- 配方没有输出：保持可运行，晋升配方本身不会因无输出被当成失败。
- 概率值为 0、低于 100% 或带概率增幅：只要属于输出或持续输出，激活核心后都强制为 100%。
- 已有终焉聚合枢纽概率绕过：放大副本已经是满概率，后续概率逻辑应保持幂等，不得再次放大数量。

## 测试方案

### 状态测试

- 无核心时倍率为 1。
- 单个空闲有效核心倍率为 10。
- 二级配方真实工作时倍率为 100。
- 一级配方真实工作时倍率为 1000。
- 配方暂停、完成或中断后恢复 10。
- 多核心同时存在时取最高值，不叠加。
- 核心断开宿主后不再提供倍率。

### 配方变换测试

- 固定物品和流体输出分别放大 10、100、1000 倍。
- 概率物品、概率流体和持续输出均变为 10000/10000。
- 输入、持续输入、EU/t 和 duration 不变。
- 同一配方构建路径只放大一次。
- 输出容量检查基于放大后数量。
- 非原初模块和直接 Tick 产出不受影响。

### 注册测试

- 两个配方类型都注册在 `gtceu` 命名空间。
- 两个类型均为 4 物品输入、0 物品输出、4 流体输入、0 流体输出。
- 机器定义绑定两个晋升类型并使用原初模块结构。
- 中英文 Lang 键和 tooltip 完整。

## 不在本次范围

- 不在 Java 中硬编码具体晋升配方材料、时长或 EU/t。
- 不修改现有原初模块的配方类型集合。
- 不放大非配方形式的能源、算力或广播产出。
- 不允许多个核心叠加超过 1000x。
