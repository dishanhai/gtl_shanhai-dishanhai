# 星律样板总成

<Column gap="15" fullWidth={true}>

<Row>
    <BlockImage id="gt_shanhai:recipe_type_pattern_buffer" scale="6" />
    <BlockImage id="gt_shanhai:recipe_type_pattern_buffer_proxy" scale="6" />
</Row>

<Column gap="2" fullWidth={true}>

星律样板总成是一种**配方类型感知的 AE2 处理样板缓冲器**。它在 GTLCore 样板总成的基础上，为每个样板槽位记录并识别对应配方的**配方类型**，使宿主机器能按类型约束配方执行。

* 本体继承 GTLCore 样板总成完整能力，并复用分页样板槽界面（默认 9×6×3 = 162 槽）。
* 每个样板槽位绑定一个配方类型 ID，放入时自动读取样板内含配方类型。
* 当宿主机器配方搜索时，星律样板总成只提供**配方类型匹配**的样板给宿主执行。
* 样板槽位数量、每行数量、每页行数和最大页数可按配置调整。

</Column>

<Column gap="2" fullWidth={true}>

## 工作原理

### 配方类型记录

星律样板总成在放入样板时，自动从样板 NBT 中读取 `gt_shanhai_recipe_type` 字段，记录为该槽位的类型约束：

```
样板槽位 0: 切割(gtceu:cutter)
样板槽位 1: 组装(gtceu:assembler)
样板槽位 2: 化学反应(gtceu:chemical_reactor)
```

### 配方执行过滤

当宿主机器运行时，如果当前配方类型与样板记录不一致，该样板对这个配方**自动过滤**（不参与搜索）。这样：

- 通用加工厂等**多配方类型机器**可在同一宿主内装入不同类型样板
- 避免"跑着切割机配方时被铸造样板提示产出"的干扰
- 不依赖样板本体手动切换配方类型，靠**静默过滤**实现隐形转接

### 与通用样板总成的区别

| 对比项 | GTLCore 通用样板总成 | 星律样板总成 |
|------|------------------|----------|
| 多类型兼容 | 手动选择 | 自动过滤 + 混装 |
| 跨类型执行 | 不支持 | 支持（虚拟注入） |
| 数据持久 | AE 样板 | NBT + 机器内存 |
| 虚拟执行 | 无 | 支持 |

</Column>

## 使用场景

### 场景 1：通用加工厂多样板

**宿主机器：** 通用加工厂（支持切割、冲压、锻造）

**配置方式：** 装入星律样板总成，在同一台机器内配置不同类型样板：
- 槽 0-20: 切割配方样板
- 槽 21-40: 冲压配方样板  
- 槽 41-60: 锻造配方样板

**运行结果：**
- 机器搜索切割配方时 —— 自动过滤，仅执行槽 0-20 的样板
- 机器搜索冲压配方时 —— 自动过滤，仅执行槽 21-40 的样板
- 机器搜索锻造配方时 —— 自动过滤，仅执行槽 41-60 的样板

### 场景 2：避免配方窜线

**普通样板总成的问题：**
- 同一台机器既运行 A 类型配方又运行 B 类型配方
- AE 计划显示的"产出预览"混杂了两种类型的预期产出
- 玩家容易把样板放错位置，导致执行错误配方

**星律样板总成的改进：**
- 每个样板槽位自动约束配方类型
- 即使样板位置放错，机器也会自动过滤，不执行
- 完全消除配方窜线风险，提高多类型机器的易用性

## 使用方式

### 基础装置

1. 把 <ItemLink id="gt_shanhai:recipe_type_pattern_buffer" /> 作为**样板输入仓室**放入多配方类型宿主机器的多方块结构。
2. 开启或重组机器后，星律样板总成自动在 GT 侧被识别为有效输入仓。
3. 通过宿主 AE2 接口向星律样板总成下单处理样板（操作方法与通用样板总成相同）。

### 通配符样板（需加载 wildcard_pattern）

星律样板总成左侧配置栏提供独立的**通配符样板**入口，横向包含 5 个母槽。放入通配符样板后，GTLCore 会按样板中的材料筛选条件展开普通处理样板；展开结果继续使用星律的配方类型过滤和库存输入拉取。

更换任意母槽时，旧展开槽的输入会先退回 ME 网络，再重新生成展开集合。相同的展开样板会自动去重。

### 槽位代理

需要跨越较远距离或规避方块碰撞时：

1. 使用 <ItemLink id="gt_shanhai:recipe_type_pattern_buffer_proxy" /> 作为**样板代理**放在结构允许位置。
2. 用数据棒右键本体复制绑定信息，再右键代理执行粘贴。
3. 代理会以本体槽位为中心向外扩展取样板，支持远程样板供应链。

### 配置调整（可选）

星律样板总成默认使用 9×6×3 配置。如果需要修改：

```
需要配置项名称来自 DShanhaiConfig.java
- DShanhaiConfig.RECIPE_TYPE_PATTERN_BUFFER_PATTERN_PER_ROW（默认 9）
- DShanhaiConfig.RECIPE_TYPE_PATTERN_BUFFER_ROWS_PER_PAGE（默认 6）
- DShanhaiConfig.RECIPE_TYPE_PATTERN_BUFFER_MAX_PAGES（默认 3）
```

在 Minecraft config 目录编辑配置文件后重启游戏，新放置的星律样板总成即采用新配置。

## 虚拟跨类型执行

星律样板总成支持**虚拟跨配方类型执行**：样板携带的配方类型**无需宿主机器原生支持**，即可在宿主内虚拟直跑。

* **虚拟注入** — 样板配方作为完整 GT 配方通过 ME 样板处理链挂载并入宿主执行队列，执行不依赖宿主原生配方类型。
* **跨类型直跑** — 例如给只支持切割的宿主装入压缩配方样板，也能正常执行产出。
* **多类型混装** — 同一宿主内可同时装入不同配方类型的样板，一起参与执行，无需统一类型。
* **全宿主生效** — 任何能接入星律样板总成的宿主（含 GTCEu / GTLCore 原生机器）都获得该能力。

### 切换模式（可选）

行为由配置项 `DShanhaiConfig.recipeTypePatternSwitchMode` 控制：

* **VIRTUAL_ACTIVE_TYPE（默认）** — 纯虚拟注入。不校验宿主是否原生支持，不改宿主运行模式，直接虚拟执行，支持跨类型与多类型混装。
* **PROGRAMMABLE_HATCH_REQUIRED** — 严格模式。要求山海可编程仓随主机成型并成功切换宿主配方类型才执行虚拟样板，且要求样板类型唯一；无可编程仓时不运行。

### 当前版本边界

* **不生成 ticket/slot lock** — 样板执行沿用宿主原生 GT 配方逻辑，不额外加锁。
* **无并行倍增** — 并行上限由宿主机器决定，星律样板总成不额外调整。

## 常见问题

### Q: 能否在 GTCEu 原生机器（如通用加工厂）里用？
**A:** 是的。只要该机器支持 ME 样板缓冲器能力（`IMPORT_ITEMS` + `IMPORT_FLUIDS`），星律样板总成即可作为仓室接入。

### Q: 样板放错位置了怎么办？
**A:** 机器会自动过滤类型不匹配的样板。放错位置不会导致配方执行，但建议整理后再次操作。

### Q: 能否混合多个样板总成（星律 + 通用）？
**A:** 可以。两种样板总成在同一宿主内会一起参与配方搜索。星律样板总成做类型过滤，通用样板总成不做过滤，两者互补。

### Q: 为什么我的宿主机器没有识别到星律样板总成？
**A:** 确认：
1. 机器多方块已正确成型
2. 星律样板总成放在宿主允许的仓室位置（通常是结构外框 / 特定槽位）
3. 游戏日志中无 "ERROR" 关键字
如仍有问题，报告详细日志。

### Q: 样板类型是如何记录的？
**A:** AE2 模式编码 UI 中生成的样板会自动包含 `recipe_type` 信息（由 GT 贡献）。星律样板总成读取这个信息，无需手动填写。手写样板则需确保包含 `gt_shanhai_recipe_type` NBT 字段。
